# Per-Launch Lightning Wallets

Replace the fixed pool of pre-registered PumpPortal Lightning wallets with a fresh wallet generated per launch at creation time. The Lightning wallet IS the escrow for that launch — contributor SOL lands directly in it, PumpPortal launches the token from it, and creator fees accrue to it.

## Core idea

Today, two wallets are involved per Pump.fun Lightning launch:
- An **escrow wallet** (generated by `create-launch-pumpfun`, private key encrypted in DB) that receives contributor SOL.
- A **shared custodial wallet** picked from the Railway/DB pool that PumpPortal funds and uses to sign the create+buy.

After this change, both roles collapse into one **per-launch Lightning wallet** generated by `https://pumpportal.fun/api/create-wallet` at creation time. PumpPortal returns `walletPublicKey` + `privateKey` (bs58) + `apiKey`; we encrypt and store all three on the launch row, and contributors send SOL directly to that pubkey.

## Schema changes (`launches`)

Add three nullable columns (nullable so existing rows keep working):
- `lightning_wallet_public_key text`
- `lightning_wallet_encrypted_private_key text`  — same AES‑256‑GCM `iv:authTag:ciphertext` format used for `escrow_wallet_encrypted_private_key`. Plaintext = the bs58‑decoded 64‑byte secret key, hex‑encoded (matches `decryptEscrowKey` expectations so existing decrypt code is reused unchanged).
- `lightning_wallet_encrypted_api_key text` — AES‑256‑GCM of UTF‑8 hex of the api key string.

We **also reuse** the existing `escrow_wallet_public_key` column. For new pumpfun launches we set it to the Lightning wallet pubkey so the contribution flow, `LaunchPage`, refund tooling, accounting/recovery UIs, and the validate/contribute edge functions continue to verify destination addresses with zero changes.

Likewise `escrow_wallet_encrypted_private_key` is set to the encrypted Lightning private key, so the executor's existing `escrowKeypair` decode + sweep + refund paths just work.

Old `pumpportal_wallet_pubkey` keeps its meaning (assignment from the pool) for any in‑flight legacy launches; new launches simply leave it null.

## `create-launch-pumpfun/index.ts` changes

After generating the mint keypair (the local escrow keypair generation can stay or be removed — keeping it is harmless and serves as a fallback if PumpPortal is down), and before metadata/IPFS verification:

1. `GET https://pumpportal.fun/api/create-wallet` with a 15s timeout and one retry.
2. Validate the response shape: `walletPublicKey` (base58, 32–44 chars), `privateKey` (base58 secret), `apiKey` (non-empty string).
3. Decode `privateKey` from bs58 to a 64‑byte Uint8Array, hex‑encode, then `encryptKey(...)` with `ESCROW_ENCRYPTION_KEY`.
4. Encrypt `apiKey` (hex‑encoded UTF‑8) the same way.
5. On the insert:
   - `escrow_wallet_public_key = walletPublicKey`
   - `escrow_wallet_encrypted_private_key = encryptedLightningPrivKey`
   - `lightning_wallet_public_key = walletPublicKey`
   - `lightning_wallet_encrypted_private_key = encryptedLightningPrivKey`
   - `lightning_wallet_encrypted_api_key = encryptedApiKey`
   - keep mint fields as today.
6. If `create-wallet` fails after retry, return 503 ("PumpPortal wallet provisioning unavailable") so the user can retry — no DB row is created.

The local ed25519 escrow keypair generation block can be removed since the Lightning wallet replaces it; we leave the bs58 + AES helpers alone.

## Contribution flow

No code changes in `validate-contribution`, `contribute`, `LaunchPage`, or wallet UIs — they all read `launch.escrow_wallet_public_key`, which now points at the Lightning wallet.

## Executor (`executePumpfunLightning.ts`)

Detect "new model" launches with `if (launch.lightning_wallet_public_key)`:

- **No custodial-wallet pool lookup.** Skip `resolveLaunchWallet` and the `pumpportal_wallet_pubkey` persistence block entirely.
- **No escrow→custodial funding step.** SOL is already in the Lightning wallet, which is the same wallet PumpPortal uses to sign. Skip `fundCustodialWallet`.
- Build a `PumpPortalWallet`-shaped object from the launch row:
  - `pubkey` = `lightning_wallet_public_key`
  - `publicKey` = `new PublicKey(pubkey)`
  - `secretKey` = `decryptEscrowKey(launch.lightning_wallet_encrypted_private_key)`
  - `apiKey` = AES‑decrypt `lightning_wallet_encrypted_api_key`
  - `slot` = -1 (sentinel; not used for anything except logs)
- Use this wallet's `apiKey` for the `https://pumpportal.fun/api/trade?api-key=...` call. Body unchanged (action=create, mint=bs58 secret, etc.).
- The processing-fee, basis-points, mint validation, polling, refund/cancel paths all stay exactly as today (they operate on the escrow keypair, which now is the Lightning wallet).
- After landed=succeeded, `sweepTokensToWallet(connection, mint, escrowKeypair.publicKey, wallet)` becomes a self‑sweep into the same wallet — replace with: skip token sweep if `wallet.publicKey.equals(escrowKeypair.publicKey)` (tokens are already in the right place). Same for `sweepSolToWallet` residual sweep.
- Custodial lock key = the Lightning wallet pubkey, so per-launch operations are naturally serialized only against themselves (no cross-launch contention since each launch has its own wallet).

Legacy launches (those with `pumpportal_wallet_pubkey IS NOT NULL` or `lightning_wallet_public_key IS NULL`) keep the existing code path unchanged. The Railway env var pool + DB `lightning_wallets` table stay as the fallback for them. No edits to `pumpportalWalletPool.ts`.

## Fee claiming (`distributor/src/claimPumpfunFees*.ts`)

Same branch: if `launch.lightning_wallet_public_key` is set, decrypt the per‑launch api key/secret from the launch row instead of resolving from the pool. Otherwise keep the legacy path. Fees accrue to the Lightning wallet (which is the escrow), so `recipient` for the fee claim becomes the Lightning wallet itself — no extra sweep needed for fees that already land there.

## Refund / sweep / recovery

Untouched. They use `escrow_wallet_encrypted_private_key`, which on new launches decrypts to the Lightning wallet's secret key.

## Operational notes

- PumpPortal `create-wallet` is unauthenticated and returns instantly; failures are rare but we treat them as a hard 503 so we never write a partial launch row.
- The Lightning wallet starts at 0 SOL. PumpPortal Lightning's create+buy requires the wallet to have ≥ buy amount + ATA rent + protocol fee + tx fee. Since contributors fund the wallet directly during the contribution window (min 0.3 SOL pool), this is naturally satisfied. The processing-fee debit (escrow→treasury) still runs first and uses the Lightning wallet as the source.
- `CUSTODIAL_FUNDING_BUFFER_LAMPORTS` (0.025 SOL) is no longer needed for new launches — the buffer existed to cover the escrow→custodial transfer + custodial-side overhead. With the wallets unified, the existing reserve math (ATA reserve + tx fee) already covers everything; the saved 0.025 SOL goes into the buy.
- 20‑wallet generation step from the previous plan is obsolete and not needed.

## Open question

Should we keep generating the local ed25519 escrow keypair as a fallback (and only overwrite the columns once `create-wallet` succeeds), or remove it entirely? Recommend **remove** for clarity — if PumpPortal can't issue a wallet we can't launch anyway, so failing loudly at create-time is preferable to creating an unusable launch row.
